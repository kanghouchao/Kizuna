# 現行業務フロー一覧

現行業務システムの業務を Kizuna が引き受けるための材料として、「誰が、何をきっかけに、どの処理を行い、どの業務結果が残るか」を業務語彙のみで記録する。出典は #362（本文と回答コメント）。移行ロードマップ地図 #367 の材料であり、裁定票 #368〜#370 の入力になる。

方針（#367 の Notes に準拠）:

- 旧システムの画面名・内部用語・ディレクトリ構成・実装方式は記録しない。把握するのは**機能・データ・役割**の 3 視点のみ。
- 本質を先に、細部は後で。各フローの詳細手順は裁定後に追加の Q&A で補う。
- データの詳細は data-model.md（#363）、利用者・権限の詳細は actors-and-access.md（#364）が持つ。本書は業務の流れだけを持つ。

## 各フロー

各フローは「開始 → 完了」の業務としての本質のみを記す。

### 1. 組織・店舗・社内アカウント管理（横断）

- 開始 → 完了: 管理担当者が店舗・利用者・権限を登録・変更する → 利用者が許可された機能を、担当店舗の範囲だけで利用できる。
- 失ってはいけない要点（確認済み）: 機能権限・担当店舗・精算対象店舗の組合せで利用範囲が決まる。
- 特別権限・例外の棚卸しを含む詳細は actors-and-access.md（#364）へ。

### 2. キャスト在籍・プロフィール管理（核心域）

- 開始 → 完了: 店舗担当者が入店・再在籍・プロフィールを登録する → 同じキャストの所属・在籍状態・内部情報・公開情報が対応付いている。
- 失ってはいけない要点（確認済み）: 内部情報と公開情報を別物として区別しつつ、同一キャストとして対応付ける。
- 注意（確認済み）: 同じ意味の値が複数箇所に存在し、正本が一つに定まっていない（詳細は data-model.md）。

### 3. 出勤希望から公開シフトまで（核心域）

- 開始 → 完了: キャストが希望を提出し、店舗が確認する → 希望・承認済みシフト・当日実績・公開状態が区別されて残る。
- Kizuna 既定: 出勤希望の提出+店舗承認は黄金フロー③（#329）、確定シフトの変更申請は #330。
- 差分（確認済み）: 現行は希望・承認・実績・公開の処理が複数機能に分散している。Kizuna では状態と正本を一つに決める必要がある。

### 4. 顧客受付・顧客台帳（核心域）

- 開始 → 完了: 店舗が電話番号等で顧客を検索する → 連絡先・利用履歴・注意事項・連絡可否を次回受付で確認できる。
- 失ってはいけない要点（確認済み）: 電話受付の結果（通話自体はシステム外）を顧客・受注情報に登録し、次回受付へ引き継ぐ。

### 5. Web 予約申請（核心域）

- 開始 → 完了: 会員が店舗・日時・キャスト・サービス内容を送信する → 仮予約として受け付け、店舗確認後に確定する。
- Kizuna 既定: 会員の予約申請→店舗確定は黄金フロー④（#334）。
- 差分（確認済み）: 現行は Web 上「仮予約」と案内しつつ、内部では早い段階で既存の受注作成フローに合流する。Kizuna で申請と確定受注を分ける場合、旧受注との対応付けが必要。

### 6. 店舗受付から受注完了まで（核心域）

- 開始 → 完了: 店舗担当者が Web 申請や電話受付から受注を作成する → 顧客・店舗・キャスト・日時・料金・割引・ポイント・移動担当・実績時刻が一つの受注として追跡できる。
- 失ってはいけない要点（確認済み）: 受注は売上・給与・ポイント等すべての根拠。ただし関連処理がすべて同時に完了する構造ではない。
- Kizuna 既定: Order 集約は金額と人数を持つ方針（#321 裁定、未実装 — 現行の `Order` / `OrderCreateRequest` に金額・人数フィールドはない）。移動担当（配車）は Kizuna 未対応 — #369 の裁定対象。

### 7. 精算・給与・集計（金流域）

- 開始 → 完了: 店舗・経理担当者が完了受注を締める → 売上・回収額・キャスト給与・移動担当精算・店舗間精算が受注と照合できる。
- 失ってはいけない要点（確認済み）: 完了受注との照合可能性と、算出根拠の保持（詳細は data-model.md）。
- Kizuna: 金流域は未対応 — #369 の裁定対象。Excel / CSV 出力・印刷の外部接点あり（後述）。

### 8. 会員登録と顧客情報の関連付け（核心域）

- 開始 → 完了: 利用者が会員登録する → 会員本人と店舗側顧客記録が関連付き、履歴・ポイント・予約を利用できる。
- Kizuna 既定: 会員登録+会員コード発行（#331）、店舗側コード読取→顧客台帳の紐づけ（#333）、LINE ログイン（#332）。
- 差分（確認済み）: 現行は主に電話番号で照合し、安定した会員―顧客関連 ID がない。重複・電話番号変更・関連付け解除・顧客統合の扱いが必要（詳細は data-model.md）。

### 9. ポイント・会員特典（核心域）

- 開始 → 完了: 受注完了・キャンペーン・紹介等を契機に処理する → 残高に加え、増減理由・利用・取消・期限・ランク・特典の根拠が追跡できる。
- Kizuna 既定: 会員ポイント台帳+オーダー完了処理は黄金フロー⑤（#335）、ポータル側画面は #336。Customer.points は廃止方針（#321 裁定、未実施 — 現行コード・スキーマ（`03-crm-schema.yaml`）には points が残っている）。
- 差分（確認済み）: 現行は残高・期限別ポイント・利用履歴・ランク更新・期限案内が別処理で、受注完了と同時にすべて確定する構造ではない。ルール詳細（期限・失効・消費順）は data-model.md。

### 10. 公開サイトへの情報掲載（公開域）

- 開始 → 完了: 店舗・掲載担当者が公開内容を更新する → 店舗・キャスト・出勤・記事・ランキング・口コミ・写真付き投稿が適切な公開範囲・期間・順序で表示される。
- Kizuna: 店舗・キャスト・出勤・メニュー等の基本的な公開表示は対応済み（`frontend/src/_pages/store-site`、`/tenant/config` `/casts` `/shifts` の public API）。記事・ランキング・口コミ・写真付き投稿は未対応 — #370 の裁定対象（#320 でエリア/サイトのコンポーネント化は後置と裁定済み）。#165（CMS コンテンツ編集）が隣接。

### 11. 写真付き投稿・外部サービス連携（公開域）

- 開始 → 完了: キャストまたは掲載担当者が投稿する → 投稿・画像・対象キャスト・配信先・外部識別子・成功/失敗が追跡できる。
- 失ってはいけない要点（確認済み）: 外部識別子による重複判定と、送信結果（成功・失敗）の追跡。メールが投稿・外部連携の入口として使われる。
- Kizuna: 未対応 — #370 の裁定対象。#171（メディア管理）が隣接。

### 12. 顧客連絡・口コミ・アンケート（公開域）

- 開始 → 完了: 店舗または自動処理が対象者へ送信する → 対象者・内容・送信状態・回答・公開/承認状態を確認できる。
- Kizuna: #172（通知システム）が隣接するが、口コミ・アンケートの承認フローは未対応 — #370 の裁定対象。

### 13. 定期処理・監査（横断）

- 開始 → 完了: 時刻または業務イベントを契機に自動実行する → 対象・処理結果・失敗・再実行・実行元を追跡できる。
- 失ってはいけない要点（確認済み）: 失敗と再実行の追跡可能性。
- Kizuna: 未対応 — #370 の裁定対象。

## システム外との接点（確認済み）

- 電話受付・営業電話: 通話自体はシステム外。結果を顧客・受注情報へ登録する。
- LINE: 連絡先案内や ID 保持はあるが、LINE 認証ではなく、ログイン・会話履歴の一元管理は確認できない。
- メール: 写真付き投稿や外部サービス連携の入口。
- Excel / CSV: 受注表・シフト・精算明細の Excel 出力、会計向け CSV 出力。
- 印刷: 在籍一覧・プロフィール・精算明細・アンケート等。

これらの利用頻度と、移行後も外部運用として残す範囲は業務確認待ち。
