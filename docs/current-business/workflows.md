## 各フロー

各フローは「開始 → 完了」の業務としての本質のみを記す。

### 1. 組織・店舗・社内アカウント管理（横断）

- 開始 → 完了: 管理担当者が店舗・利用者・権限を登録・変更する → 利用者が許可された機能を、担当店舗の範囲だけで利用できる。
- 失ってはいけない要点（確認済み）: 機能権限・担当店舗・精算対象店舗の組合せで利用範囲が決まる。
- 特別権限・例外の棚卸しを含む詳細は actors-and-access.md（#364）へ。

### 2. キャスト在籍・プロフィール管理（核心域）

- 開始 → 完了: 店舗担当者が入店・再在籍・プロフィールを登録する → 同じキャストの所属・在籍状態・内部情報・公開情報が対応付いている。
- 失ってはいけない要点（確認済み）: 内部情報と公開情報を別物として区別しつつ、同一キャストとして対応付ける。
- 注意（確認済み）: 同じ意味の値が複数箇所に存在し、正本が一つに定まっていない（詳細は data-model.md）。

### 3. 出勤希望から公開シフトまで（核心域）

- 開始 → 完了: キャストが希望を提出し、店舗が確認する → 希望・承認済みシフト・当日実績・公開状態が区別されて残る。
- Kizuna 既定: 出勤希望の提出+店舗承認は黄金フロー③（#329）、確定シフトの変更申請は #330。
- 差分（確認済み）: 現行は希望・承認・実績・公開の処理が複数機能に分散している。Kizuna では状態と正本を一つに決める必要がある。

### 4. 顧客受付・顧客台帳（核心域）

- 開始 → 完了: 店舗が電話番号等で顧客を検索する → 連絡先・利用履歴・注意事項・連絡可否を次回受付で確認できる。
- 失ってはいけない要点（確認済み）: 電話受付の結果（通話自体はシステム外）を顧客・受注情報に登録し、次回受付へ引き継ぐ。

### 5. Web 予約申請（核心域）

- 開始 → 完了: 会員または**未ログインの来訪者**が店舗・日時・キャスト・サービス内容（来訪者は連絡先を含む）を送信する → 仮予約として受け付け、店舗確認後に確定する。
- 失ってはいけない要点（確認済み）: 予約入口は会員に限らない（#375 の監査で確認された事実）。ゲスト予約と会員予約を両方受け付け、必要に応じて後から顧客・会員へ安全に関連付ける。
- Kizuna 既定: 会員の予約申請→店舗確定は黄金フロー④（#334）。
- 増補前提（#375 監査由来 — 裁定待ち、#334 コメント参照）: #334 を唯一の予約入口としない。
- 差分（確認済み）: 現行は Web 上「仮予約」と案内しつつ、内部では早い段階で既存の受注作成フローに合流する。Kizuna で申請と確定受注を分ける場合、旧受注との対応付けが必要。

### 6. 店舗受付から受注完了まで（核心域）

- 開始 → 完了: 店舗担当者が Web 申請や電話受付から受注を作成する → 顧客・店舗・キャスト・日時・料金・割引・ポイント・移動担当・実績時刻が一つの受注として追跡できる。
- 失ってはいけない要点（確認済み）: 受注は売上・給与・ポイント等すべての根拠。ただし関連処理がすべて同時に完了する構造ではない。
- Kizuna 既定: Order 集約は金額と人数を持つ方針（#321 裁定、未実装 — 現行の `Order` / `OrderCreateRequest` に金額・人数フィールドはない）。移動担当（配車）は Kizuna 未対応 — #369 の裁定対象。

### 7. 精算・給与・集計（金流域）

- 開始 → 完了: 店舗・経理担当者が完了受注を締める → 売上・回収額・キャスト給与・受注ごとの資金受渡し（移動担当精算）が受注と照合できる。
- 失ってはいけない要点（確認済み）: 完了受注との照合可能性と、算出根拠の保持（詳細は data-model.md）。
- 訂正（確認済み — 2026-07-17 回答）: 「移動担当精算」は移動担当者の報酬計算ではなく、受注ごとの預かり金・釣銭・前払い・ホテル代・交通関連費用・キャスト給与の資金受渡し確認処理。移動担当者の報酬算式は現行に確認できない（報酬管理を置く場合は移行ではなく新要件）。店舗間精算は独立した台帳・専用フローを確認できず、確認済み事実から除外して業務確認待ちへ変更（詳細は data-model.md「精算・資金受渡しと店舗間精算」）。
- Kizuna: 金流域は未対応 — #369 の裁定対象。Excel / CSV 出力・印刷の外部接点あり（後述）。

### 8. 会員登録と顧客情報の関連付け（核心域）

- 開始 → 完了: 利用者が会員登録する → 会員本人と店舗側顧客記録が関連付き、履歴・ポイント・予約を利用できる。
- Kizuna 既定: 会員登録+会員コード発行（#331）、店舗側コード読取→顧客台帳の紐づけ（#333）、LINE ログイン（#332）。
- 差分（確認済み）: 現行は主に電話番号で照合し、安定した会員―顧客関連 ID がない。重複・電話番号変更・関連付け解除・顧客統合の扱いが必要（詳細は data-model.md）。

### 9. ポイント・会員特典（核心域）

- 開始 → 完了: 受注完了・キャンペーン・紹介等を契機に処理する → 残高に加え、増減理由・利用・取消・期限・ランク・特典の根拠が追跡できる。
- Kizuna 既定: 会員ポイント台帳+オーダー完了処理は黄金フロー⑤（#335）、ポータル側画面は #336。Customer.points は廃止方針（#321 裁定、未実施 — 現行コード・スキーマ（`03-crm-schema.yaml`）には points が残っている）。
- 差分（確認済み）: 現行は残高・期限別ポイント・利用履歴・ランク更新・期限案内が別処理で、受注完了と同時にすべて確定する構造ではない。ルール詳細（期限・失効・消費順）は data-model.md。

### 10. 公開サイトへの情報掲載（公開域）

- 開始 → 完了: 店舗・掲載担当者が公開内容を更新する → 店舗・キャスト・出勤・記事・ランキング・口コミ・写真付き投稿が適切な公開範囲・期間・順序で表示される。
- Kizuna: 店舗・キャスト・出勤・メニュー等の基本的な公開表示は対応済み（`frontend/src/_pages/store-site`、`/store/config` `/casts` `/shifts` の public API）。記事・ランキング・口コミ・写真付き投稿は未対応 — #370 の裁定対象（#320 でエリア/サイトのコンポーネント化は後置と裁定済み）。#165（CMS コンテンツ編集）が隣接。

### 11. 写真付き投稿・外部サービス連携（公開域）

- 開始 → 完了: キャストまたは掲載担当者が投稿する → 投稿・画像・対象キャスト・配信先・外部識別子・成功/失敗が追跡できる。
- 失ってはいけない要点（確認済み）: 外部識別子による重複判定と、送信結果（成功・失敗）の追跡。メールが投稿・外部連携の入口として使われる。
- メール入口の実態（確認済み — 2026-07-17 回答）: メール経由の投稿・配信は現在も有効な経路。メール受信を入口にする処理と、外部媒体から投稿を取得する処理の両方があり、取得後に別の配信先へメールで送る経路も残る。キャスト本人のメールアドレス・媒体ごとの投稿先アドレス・送信元/返信先・送信結果を扱う。キャストポータルを投稿画面として採用しても、全配信先が API 等へ切り替わるまでメール連携自体は削除できない。ただしこれはキャストが毎回入力する項目ではなく、内部の外部連携設定として管理できる（移行元提案 — #370 候補 2 の入力）。
- Kizuna: 未対応 — #370 の裁定対象。#171（メディア管理）が隣接。

### 12. 顧客連絡・口コミ・アンケート（公開域）

- 開始 → 完了: 店舗または自動処理が対象者へ送信する → 対象者・内容・送信状態・回答・公開/承認状態を確認できる。
- Kizuna: #172（通知システム）が隣接するが、口コミ・アンケートの承認フローは未対応 — #370 の裁定対象。

### 13. 定期処理・監査（横断）

- 開始 → 完了: 時刻または業務イベントを契機に自動実行する → 対象・処理結果・失敗・再実行・実行元を追跡できる。
- 失ってはいけない要点（確認済み）: 失敗と再実行の追跡可能性。
- Kizuna: 未対応 — #370 の裁定対象。

### 14. 広告費・集客効果管理（金流域）

2026-07-17 の #362 追加回答で確認された独立業務（当初の 13 フロー表に含まれていなかった）。

- 開始 → 完了: 広告費管理の権限を持つ担当者が、店舗・掲載月ごとの広告費入力、前月データの引継ぎ、または当月実績の確認をきっかけに、媒体別の件数・費用を確認・入力し（登録情報・受注情報からの読込を含む）、区分別・媒体別・全体の金額を集計して保存する → 確定権限を持つ管理者が対象店舗・対象月を月単位でロックし、記録が集客・採用の効果と照合できる。
- 残る記録の項目一覧は data-model.md の分類表「広告費・集客効果」を正とする（本書には重複記載しない）。
- 失ってはいけない要点（確認済み）: 月単位の確定とロック（解除あり — 確定後は通常の保存を止める）。効果確認 — 店舗・媒体別の反響件数、営業広告費と総受注数・新規受注数の比較（受注 1 件当たり/新規 1 件当たりの広告費）、採用広告費と入店人数・媒体経由人数の比較（採用 1 人当たりの費用）、採用後 2〜4 か月の定着状況、月次・年次収支集計への反映。
- 注意（確認済み）: 掲載内容・外部媒体への送信成否・顧客への送信結果までを広告費と一体で追跡する構造ではなく、媒体別費用から売上までを結び付ける完全な ROI 管理でもない。
- 各フローとの関連付けに店舗・媒体・対象月を共通の手掛かりとして保持する — Kizuna 側で必要になる要件（#362 回答の提言。確認済みの現行事実ではない）。
- 関連: 外部媒体への掲載連携はフロー 11、自社公開サイト内の掲載管理はフロー 10、顧客への案内配信はフロー 12 が持つ。
- Kizuna: 未対応 — #369 の裁定対象（月次確定・ロックと精算締めの関係、収支集計との統合を含む）。

### 15. 応募・面接・採用（横断・在籍前段）

#375 の整合性監査で存在が確認された独立業務（当初の 13 フロー表に含まれていなかった）。

- 開始 → 完了: 応募を受け付ける → 履歴書・面接・採用判断を経て、採用者がキャスト在籍登録（フロー 2）へ進む。
- 記録内容（確認済み — 2026-07-17 回答）: 応募者が履歴書情報を入力する Web 導線がある。記録できる主な内容は本人・連絡先・住所・本人確認書類・経歴・希望条件・面接日・面接担当者・面接メモ・確認項目・入店日・応募元媒体・紹介/スカウト等。ただし電話・Web・媒体を「応募受付チャネル」として統一管理する独立した応募データは無い。
- 状態の欠落（確認済み）: 状態は実質「入店」と「未入店」が中心で、応募受付・選考中・面接済・採用・不採用・辞退を区別する一貫した選考ステータスは確認できない（「未入店」だけでは不採用・辞退・保留を判別できない）。記録は店舗と面接担当者に結び付くが、本部選考・店舗選考・承認者を区別する承認フローも確認できない。未入店者も一覧と履歴書情報に残り、一定期間後の自動削除・匿名化の処理は確認できない — 保持期間は現行仕様ではなく、業務・個人情報管理上の方針として新たに決める。
- 採用効果との接続（確認済み）: 応募元媒体は記録でき、媒体別の入店人数・採用 1 人当たり費用・採用後の定着状況を広告費集計と照合する処理がある — 応募元媒体は移行対象の重要データ（フロー 14 / #369 候補 4 指標③）。
- 移行元提案: 「応募者」を在籍キャストと分け、応募受付経路・応募元媒体・選考状態・面接記録・採否/辞退理由・担当店舗・担当者を持たせ、採用時にキャストへ引き継ぐ形が現行業務に最も近い（#370 候補 5 の入力）。
- キャスト招待（#327 実装済み）との接点に注意。
- Kizuna: 未対応 — #370 裁定済み（「移行する」を採択）。選考の最終責任者と応募者情報の保持期間は業務確認待ち。

## システム外との接点（確認済み）

- 電話受付・営業電話: 通話自体はシステム外。結果を顧客・受注情報へ登録する。
- LINE: 連絡先案内や ID 保持はあるが、LINE 認証ではなく、ログイン・会話履歴の一元管理は確認できない。
- メール: 写真付き投稿や外部サービス連携の入口。
- Excel / CSV: 受注表・シフト・精算明細の Excel 出力、会計向け CSV 出力。
- 印刷: 在籍一覧・プロフィール・精算明細・アンケート等。

移行後の外部運用境界（移行元提案 — 2026-07-17 回答）: 電話は通話を外部のまま結果だけを顧客・予約・受注へ記録する / Excel・CSV・印刷は出力手段として残すが正本は Kizuna 側に置く / LINE は新しい連携を実装するまで外部連絡手段として扱う（現行は連絡先案内が中心で会話履歴の一元管理は無い） / メール投稿は配信先ごとの代替手段が確認できるまで外部連携として残す。これらの利用頻度は業務確認待ち。

2026-07-17 の spec 化前 Q&A で、応募・採用の記録内容と外部運用境界の提案は本文へ反映済み。回答者による現行コードとの再照合は実施済みだが、**現場担当者による確認は未実施**のため、次は業務確認待ちのまま残す（回答の指示）:
