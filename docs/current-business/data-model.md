# 現行業務の重要データと不変条件

現行業務システムの業務を Kizuna が引き受けるための材料として、重要データの業務上の意味・関係・状態・整合条件を業務語彙のみで記録する。出典は #363（本文と回答コメント）。移行ロードマップ地図 #367 の材料であり、裁定票 #368 / #369 / #371 の入力になる。

方針（#367 の Notes に準拠）:

- 旧システムのテーブル名・内部用語・実装方式は記録しない。ここに書くのは論理データ構造と業務ルールであり、Kizuna の物理テーブル設計・多重度は決めない。
- 本質を先に、細部は後で。詳細は裁定後の追加 Q&A で補う。
- 業務の流れは workflows.md（#362）、利用者・権限は actors-and-access.md（#364）が持つ。本書はデータだけを持つ。

最終更新: 2026-07-17（#363 の追加回答: コース構造・受注金額・ポイント加算・給与の反映時点）

## 規模感（匿名化テストデータの参考値）

- 顧客 約 30 万件 / 受注 約 100 万件・受注明細 約 22 万件 / キャスト 約 4,000 件 / 会員候補 約 4 万件（ログイン可能会員数とは一致しない）
- 写真付き投稿は別管理のため件数未確認。本番件数ではないため容量設計の参考値としてのみ扱う。

### 顧客と会員の関係（確認済み）

- 顧客と会員は別に管理され、主に電話番号で照合される。安定した会員―顧客の関連 ID はない。
- 重複顧客を一つにまとめ、受注・連絡先・履歴・ポイント・特典を統合先へ関連付け直す「顧客統合」機能が現行にある。
- したがって Kizuna 側で必要になるのは: 会員と顧客の明示的な関連 / 電話番号変更 / 複数連絡先 / 関連付け・解除 / 重複候補の確認 / 顧客統合履歴。
- 重複・紐付け誤りの本番割合は業務確認待ち。

### コース（サービス項目）と店舗別提供条件（確認済み）

受注金額・キャスト掲載・店舗の営業範囲を決める中核概念。2026-07-17 の追加回答で確認。

- コースは店舗・サービス種別・名称・所要時間を持つ。顧客向け料金は時間帯別、キャスト給与は時間帯・ランク別に設定でき、キャスト個人別の給与上書きがある。割引・指名・特別区分・延長単位など、料金・給与計算を分岐させる区分を持つ。
- **全店共通の正本は存在しない**。店舗ごとに別設定として保持され、同じ内容に見えるコースも店舗が違えば別設定。更新はその店舗のコース・料金・給与設定を一括で置き換える方式で、**有効期間・版管理がない** — 過去時点の設定を後から再現できない。
- free 区分は割引・指名区分で計算経路を変えるための区分であり、「顧客料金 0 円のサービス」を意味しない。実際の 0 円受注は受注金額・割引側で表現される。
- キャスト別の「提供可能コース一覧」は独立して存在しない。該当する給与設定が 0 の場合に受注画面で選択できないため、**給与設定が事実上の提供可否を兼ねている**。この可否が公開プロフィールへ自動反映される処理はない — Kizuna では「提供可否」と「公開表示」を別項目として明示する必要がある。
- 店舗が受注できるサービス範囲は、店舗ごとに登録されたサービス種別・時間・料金・給与の組合せで決まる。内部の受注可能範囲と公開サイトの掲載範囲は一つの正本から自動生成されていない。
- 現行設計の問題（移行元の自己申告）: 店舗間の設定重複（共通と差分を区別できない）/ 有効期間・版管理なし / 提供可否が給与設定に埋没 / 受注可能範囲と公開掲載の別管理 / free の誤解されやすい命名。そのまま引き継がない判断の材料とする。

### 受注金額の構成と確定・訂正（確認済み）

- 構成要素: 基本コース料金 + 延長料金 + オプション料金 + 指名・受付区分の加算 + 場所・エリアの加算 − 割引 ± 手動調整 − ポイント利用 + クレジット利用時加算。ホテル代・交通費・釣銭は受注金額とは別の回収・精算項目として保持する。媒体は受注に記録されるが、媒体手数料を受注総額から控除する計算はない。
- 金額は受付時に計算・入力され、延長等でサービス中にも更新される。終了時刻の入力は業務上の完了であり、**財務上のロックではない**。権限のある利用者は完了後も訂正できる（終了後の減額は警告のみで禁止されない）。
- 顧客料金の精算・釣銭の精算・キャスト給与の精算はそれぞれ別の状態として管理される。
- **売上・給与・ポイントは不可分の訂正仕訳として連動しない**。受注金額・給与・付与/利用ポイントは別々に更新可能。期限付きポイント・ギフト残高の利用割当は受注更新時に無効化・再作成されるが、給与再計算・集計整合は操作側の確認に依存する。
- 集計は基本的に受注の最新値を参照し、訂正履歴を独立した台帳として残す仕組みはない。
- 受注には確定した金額・割引・ポイント・給与など主要な数値が保存されるため、マスタ変更だけで過去受注の保存値が書き換わることはない。ただしコース名称・当時の全計算条件・設定の版は保存されず、マスタ側も置換方式のため、**受注時点の算出根拠を完全には再現できない** — Kizuna では受注時点の名称・時間・料金・給与内訳・適用ルールのスナップショット保持が必要。
- 受注金額と実回収額は別項目（未回収・クレジット・別精算項目で差が出る。割引・ポイント利用は受注金額側に反映される）。実回収額が主要集計で一貫して使われているかは未確認 — 売上・入金・未収の帳票区分は業務確認待ち。

### ポイントの仕組み（確認済み）

- 通常ポイントの表示上の期限は、最終利用日から 1 年（利用がなければ登録日から 1 年）。
- 期限別ポイントには個別の期限日があり、受注・キャンペーンで付与されるものと手動設定されるものがある。
- 利用時は期限の早い期限別ポイントから消費する。
- 通常ポイントと期限別ポイントは別の仕組みで、すべてに同じ失効処理が適用されるわけではない。
- ランクは利用回数と付与ポイントの条件で更新される。
- 期限前案内は存在するが、通常ポイントを実際にゼロにする失効処理は確認できていない — 失効タイミングは業務確認待ち。

### ポイントの加算・取消（確認済み）

上記「ポイントの仕組み」の補完（2026-07-17 の追加回答）。

- 通常受注の付与は「受注金額を組織共通の基準単位で区切り、単位ごとの基準ポイントを掛ける」方式。コースごとの固定ポイントではない。ランク別付与率のサーバ側計算・設定も存在するが、終了時刻入力時の自動付与は現在無効化されており、付与欄は受注詳細で編集できる — **付与の確定時点と使用する計算規則が一つに定まっていない**。Kizuna では一つに決める必要がある。
- 手動付与は複数経路: 受注への手動付与入力 / 受注を作らないポイント付与のみ / 理由付きの顧客への直接加減算。実際の運用理由（補填・訂正・個別施策）は業務確認待ち。
- 取消は、受注金額・給与・付与/利用ポイントを 0 にして更新し、当該受注の期限付き/ギフト残高の利用履歴を無効化・再割当する。論理削除された受注は通常ポイント残高の集計対象外。ただし**受注と別台帳で付与された紹介・キャンペーン等のポイントは受注取消だけでは自動取消されない** — 取消経路ごとの扱いが統一されておらず、Kizuna では「取消仕訳」と「元取引との関連」を必須にする必要がある。
- 紹介・キャンペーンは一律ではない: 紹介特典は組織単位（被紹介者の初回有料利用を条件に紹介者・被紹介者へ付与）、ログイン特典は組織単位、来店特典は店舗情報を持てる。期限付き施策・スタンプ等は専用条件を持つ。Kizuna では各ルールに「適用組織/店舗」「条件」「期間」「重複可否」「取消方法」の明示が必要。

### キャスト給与・ボーナス（確認済み）

- 給与の基本は、店舗・時間帯・サービス種別・所要時間・キャストランクに対応するコース給与。キャスト個人別のコース給与があればそれを優先し、指名区分・延長・オプション・受付経路等の加算と個人例外額を加える。
- 取消扱い（金額 0）の受注は原則給与も 0。精算時に給与を手動訂正できる。税相当額は日ごとの給与合計と店舗・キャスト条件から別計算される。受注ベースの計算内に時給保証はない（保証制度の実運用有無は業務確認待ち）。
- **ボーナスは二種類を区別する**: ①給与に含まれる指名・延長・オプション等の加算 ②給与とは別のキャスト向けボーナス残高。②はキャスト・店舗ごとに指名・待機・緊急出勤の基準値を設定でき、権限を持つ管理側利用者が設定・手動付与する。日次の来店確認による付与もある。ただし各イベントから残高への自動付与の一連処理はコード上完全には追跡できず、設定値の存在だけで自動計算済みとは判断しない。本数・売上目標・ランキング連動は未確認 — いずれも業務確認待ち。
- 締めは日次・受注単位の精算が中心（精算画面は通常一営業日を対象）。各受注に顧客料金・キャスト給与の精算済み状態がある。キャスト本人も当日または前回出勤日の受注明細・給与合計を確認し、店舗・本人・営業日単位で確認済みを記録できる。**独立した週次/月次の給与締め期間は存在しない**。
- 照合は、取消・未成約を除く受注明細の保存済み/再計算給与を合計し、税相当額と精算済み状態を確認する方式。
- 店舗差は大きい（時間帯・コース・ランク・キャスト個人・指名・延長・オプションで変わり、一部店舗のみの分岐もある）。設定に適用開始日・終了日・版がないため、変更頻度によらず Kizuna では有効期間と変更履歴の保持が必要。

### 履歴として保持すべき根拠（確認済み）

次の履歴は現在値だけに縮約してはならない（保存年限そのものは業務確認待ち — 業務・監査方針として別途決める）。

- 受注の状態・料金・割引・ポイント・実績時刻・訂正
- 受注時点のコース内容スナップショット（名称・所要時間・料金・給与内訳・適用ルール — マスタが置換方式のため受注側で保持するしかない）
- ポイントの加算・利用・取消・期限・失効・関連受注
- 売上・給与・精算の算出根拠
- 外部サービスへ送った投稿の識別子・結果・失敗
- 顧客統合・権限変更・重要操作の履歴

### 分離して保持すべき 7 点（追加回答の帰結）

現行で混在・埋没しているため、Kizuna では次を分離して保持する必要がある（裁定票 #368 / #369 の設計入力）:

1. サービス定義
2. 店舗別提供条件（料金・給与・時間帯・ランク）
3. キャスト提供可否
4. 公開掲載
5. 受注時点スナップショット
6. 金額・ポイント・給与の訂正台帳
7. 精算状態（顧客料金・釣銭・キャスト給与）

## 状態とライフサイクル（本質のみ）

確認済み事実から導かれる状態区別のみを記す。遷移の網羅・多重度はここでは決めない（#363 受け入れ条件）。

- **予約・受注**: Web 申請（仮予約）と店舗確認後の確定受注を区別する。受注完了後も訂正がありうる。完了受注が売上・給与・ポイント・来店履歴の根拠になるが、関連処理は同時確定ではない。
- **受注金額・精算**: 受付時に計算 → サービス中の更新（延長等）→ 終了時刻入力（業務上の完了、財務ロックではない）→ 完了後の訂正。顧客料金精算・釣銭精算・キャスト給与精算は別状態。
- **コース設定**: 店舗単位の一括置換方式で版・有効期間を持たない（過去時点の設定は再現不可 — 受注側スナップショットが唯一の根拠）。
- **シフト**: 出勤希望 → 店舗承認後のシフト → 当日実績、の系列と公開状態は別軸。公開は承認・実績の後段階ではなく、承認済みシフトの公開可否として独立に変わる（承認=公開ではない — #329 の増補前提参照）。
- **会員―顧客関連**: 関連付け → 変更 → 解除 → 顧客統合による関連付け直し。現行は電話番号照合で安定した関連 ID がなく、これらすべてが履歴として残っている確認は取れていない（顧客統合の記録は現行にある）。各遷移を履歴として残すのは Kizuna 側の要件（「顧客と会員の関係」参照）。
- **ポイント**: 加算 → 利用（期限の早い順に割当）→ 取消 / 失効。通常と期限別で別の仕組み。
- **外部投稿**: 作成 → 送信試行 → 成功 / 失敗（外部識別子で重複判定、失敗と再試行を追跡。投稿ごとの配信先を保持する）。
- **広告費**: 入力（前月データの引継ぎ、または登録情報・受注情報からの媒体別件数読込）→ 区分別・媒体別・全体の集計 → 月次確定ロック（解除あり。確定後は通常の保存を止める）。詳細は workflows.md フロー 14（#362 の 2026-07-17 回答）。

## 作成・更新できる利用者とデータ範囲

機能権限・担当店舗・精算対象店舗の組合せで決まる（利用者種別ごとの棚卸しは actors-and-access.md（#364）が持つ。全権・経理・特定アカウントの例外を含む）。

機微データに対する操作として確認済みのもの:

- 完了後の受注金額の訂正: 権限のある利用者のみ（終了後の減額は警告付き）
- 精算時のキャスト給与の手動訂正: 精算権限を持つ利用者
- ポイントの手動付与・理由付き直接加減算: 管理側利用者（運用理由は業務確認待ち）
- 顧客統合: 管理側利用者（統合の記録が残る）
- キャスト向けボーナス残高の基準値設定・手動付与: 権限を持つ管理側利用者
- 広告費の月次確定ロック: 確定権限を持つ管理者（#362 の 2026-07-17 回答 — workflows.md フロー 14 として追加予定）

## 個人情報・機微情報の区分

分類表から導かれる境界は次のとおり。公開してよいのは下の「公開情報」に列挙したデータのみで、それ以外はすべて内部情報として扱う。

- **公開情報**: 公開プロフィール・公開状態の出勤情報・店舗情報・記事・ランキング・承認済み口コミ・写真付き投稿（いずれも公開範囲・期間・順序の管理付き）。
- **個人情報（内部）**: 顧客・会員の連絡先（電話・メール・LINE ID）、利用履歴、注意事項、連絡可否、会員ログイン情報、キャストの本人情報・内部管理情報。
- **機微（特に保護）**: 注意事項、給与・精算金額、顧客統合履歴、重要操作の監査履歴。
- 電話番号は個人情報であると同時に会員ログインと顧客照合の照合キーでもある（#371 の裁定対象）。

## 業務上の整合条件（9 条）と現行の実態

#363 本文の 9 条それぞれについて、回答で確認できた現行の実態。**現行ですら完全には守られていない条件がある**ことが #371（移行の不変条件裁定）の前提になる。

| # | 整合条件 | 現行の実態（確認済み） |
|---|---|---|
| 1 | Web 申請は仮予約として確定受注と区別 | 守られている（ただし内部では早期に受注作成フローへ合流 — workflows.md フロー 5） |
| 2 | ポイントは増減・期限・利用・取消・失効の根拠を保持 | 増減・期限・利用・取消の根拠は保持している。失効は通常と期限別で仕組みが異なり、通常ポイントを実際にゼロにする失効処理は未確認（業務確認待ち）— 全種別に失効根拠が存在する前提を置かない |
| 3 | 受注完了は売上・給与・ポイント・来店履歴の根拠 | 根拠ではあるが、関連処理がすべて同時に完了する構造ではない |
| 4 | 店舗・顧客・会員・キャスト・受注・外部投稿の識別関係を追跡できる | 会員―顧客だけ安定関連 ID がなく電話番号照合（本書「顧客と会員の関係」） |
| 5 | キャスト内部情報と公開プロフィールを混在させない | 分かれているが、同じ意味の値が複数箇所に存在する |
| 6 | 出勤希望・承認済みシフト・当日実績を区別 | 区別されているが、一部の連携用アカウントに表示上の例外がある |
| 7 | 権限は機能権限と担当範囲の組合せ | 基本はそのとおり。全権管理・経理・特定アカウント向けの例外がある（actors-and-access.md） |
| 8 | 状態・更新日時・実行主体・履歴を最新値だけに縮約しない | すべての重要データに変更不可の監査履歴があるわけではない。上書き・無効化・統合による関連変更がある |
| 9 | 同じ意味のデータが複数箇所にある場合、Kizuna で正本を一つに決める | 顧客と会員、キャスト内部情報と公開情報など、正本が一つに定まっていないデータがある |

## 継続利用される識別子（確認済み）

新システムでも追跡が必要な識別子。旧 ID → 新 ID の対応保持戦略は #371 の裁定対象（対応表の作成自体は #363 の対象外）。

- 公開プロフィール URL で使われるキャスト識別子
- 外部サービス上の店舗・キャスト・投稿の識別子
- 投稿の重複判定に使う外部識別子
- 受注番号
- 店舗識別子
- 会員ログインと顧客照合に使われる電話番号
- 顧客識別子・会員識別子（外部接点を持たない内部識別子だが、受注・ポイント・ログイン履歴との対応を移行後も維持するため、旧→新対応の保持対象 — 分類表の顧客/会員参照）

印刷物・領収書でどの識別子を使用しているかは業務確認待ち。

## 業務担当者への確認が必要な事項（まとめ）

- 履歴データの保存年限（分類ごと — 業務・監査方針として決める）
- 通常ポイントの実際の失効タイミング（ゼロにする処理は未確認）
- 顧客の重複・会員紐付け誤りの本番割合
- 印刷物・領収書で使用している識別子
- 写真付き投稿の件数、および本番データの件数全般（移行前の匿名集計）
- 手動ポイント付与・直接加減算の実際の運用理由（補填・訂正・個別施策の別）
- キャストの時給保証制度の実運用有無（受注ベース計算には存在しない）
- キャスト向けボーナス残高への自動付与の実態（設定画面はあるが自動処理は追跡できず）と、本数・売上目標・ランキング連動の有無
- 売上・入金・未収を帳票上どう区分しているか（実回収額の使われ方）
- コース・料金・給与設定の実際の変更頻度
