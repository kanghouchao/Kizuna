# 現行業務の利用者とアクセス範囲

現行業務システムの業務を Kizuna が引き受けるための材料として、利用者種別・機能権限・データ範囲を業務語彙のみで記録する。出典は #364（本文と回答コメント）。移行ロードマップ地図 #367 の材料であり、裁定票 #368 の入力になる。

方針（#367 の Notes に準拠）:

- 旧システムのアカウント名・数値権限・利用者 ID・実装方式は記録しない。「誰が何をできるか」と「どの範囲のデータを扱えるか」だけを扱う。
- 本質を先に、細部は後で。詳細は裁定後の追加 Q&A で補う。
- 業務の流れは workflows.md（#362）、データは data-model.md（#363）が持つ。本書は利用者と権限だけを持つ。
- #320 / #321 / #328〜#336 / #339 / #352 の実装方針は上書きしない。本書は業務事実の提供のみ。

最終更新: 2026-07-17（#364 の spec 化前 Q&A 回答: 特別アカウント 6 種の方針・緊急管理追跡の現状・操作単位・広告費/応募者情報の帰属の反映時点）

## 規模感（匿名化テストデータの参考値）

- 有効設定のある社内利用者 37 件、うち 25 件は複数店舗を担当（匿名化テストデータ上の参考値 — 本番の運用実態を表さない）。
- 実際に利用中の人数、本社・店舗・経理別の内訳は業務確認待ち。権限設計は人数確定を待つ必要はないが、アカウント移行前に本番の利用者一覧の匿名集計が必要。

## 権限モデルの本質（確認済み）

通常の社内利用者の利用範囲は、次の組合せで決まる。

- 利用できる機能（機能権限）
- 担当店舗
- 必要な場合だけ精算対象店舗
- アカウントの有効・無効状態

固定された 10 個のロールを作るのではなく、**「一人の社内 ID + 機能権限 + 担当店舗 + 必要な場合だけ精算範囲（+ 外部サービスの対象範囲）」**で表現するのが現行業務に最も近い形（確認済み）。

### 能力の統合可否

- 店舗管理・受付と当日店舗運営は同じ社内 ID で利用できる（確認済み）。ただし機能権限は分ける（移行元提案）。
- キャスト登録・公開情報管理・写真付き投稿・外部サービス運用は同じ担当者へ複数能力を付与できる（確認済み）。ただし内部情報と公開操作の権限は分ける（移行元提案）。
- 顧客・予約・受注の店舗業務は、担当店舗の範囲と組み合わせて一つの社内 ID にまとめられる（確認済み）。
- 経理・精算は通常の店舗操作とは別能力にし、対象となる精算単位を限定する（移行元提案 — 経理専用の特別権限が現行に存在することは確認済み）。
- システム管理・緊急管理は通常業務から分ける（移行元提案）。
- キャスト本人・会員本人・一般来訪者は、社内利用者とは別の本人種別として扱う（確認済み）。
- 実際の兼務パターン（誰がどの業務を兼ねているか）は業務確認待ち。

### 業務別の操作単位（移行元提案 — 2026-07-17 回答より転記）

人ごとの特別ロールを増やさず、次の能力を組み合わせる方式の提案。「標準では含めない操作」は明示的な追加付与を要する境界を示す（#382 要件 2 の初期入力）。

| 業務 | 必要な操作の例 | 標準では含めない操作 |
|---|---|---|
| 店舗受付 | 顧客・予約・受注の閲覧、登録、更新 | 会計確定、公開、全件出力 |
| 日常運用 | 出勤、担当割当、現在状況の閲覧・更新 | 経理、採用書類、公開 |
| 採用・在籍登録 | 応募者・在籍者の閲覧、登録、更新 | 公開、経理確定 |
| 公開コンテンツ | 公開プロフィール・記事・写真の登録、更新、公開 | 本人確認書類、給与、内部メモ |
| 経理 | 精算・給与・売上の閲覧、更新、確定、出力 | 採用書類、公開 |
| 広告費 | 閲覧・入力と、月次確定・解除・出力を分離 | 他の経理機能を自動付与しない |
| システム管理 | 組織・アカウント・権限管理 | 緊急操作は通常管理から分離 |
| 外部連携 | 指定された API と必要最小限のデータだけ | 対話画面、他機能、全件閲覧 |

### 応募者情報のアクセス範囲（2026-07-17 回答）

- 現行の応募者・面接情報は機能権限と担当店舗の範囲で制御される（確認済み）。保存内容には氏名・連絡先・住所・本人確認書類・面接情報・社内メモ等、特に慎重な扱いが必要な情報が含まれる。
- 妥当なアクセス範囲（移行元提案 — #370 候補 5 / #382 要件 2 の入力）: 担当店舗または担当組織の採用・在籍登録担当 = 閲覧・登録・更新 / 全体の採用管理が業務上必要な責任者 = 必要な場合のみ店舗横断の閲覧・承認 / システム管理者 = 保守上のアクセスに限定し通常業務では閲覧しない。店舗受付・経理・公開コンテンツ担当・外部連携用 ID には標準では付与しない。
- 現行は「採用」「不採用」「辞退」「保留」の状態と、それぞれの保存期間が十分に分かれていない（確認済み）— 新システムで明示的な状態にし、閲覧範囲と保存・削除方針を別途定める（workflows.md フロー 15 参照）。

## 特別アカウントの棚卸し

現行の例外 6 種の業務理由と種別ごとの引き継ぎ方針（2026-07-17 回答で確定 — 仕組みの存在は確認済み、業務理由欄はコードから確認できた範囲の整理〔種別 4 のように理由を確認できないものを含む〕、引き継ぎ方針は移行元提案。#368 裁定済み、増補 spec #382 が採択）:

| 例外（確認済み） | 業務理由（確認できた範囲） | 引き継ぎ方針（移行元提案） |
|---|---|---|
| 全体を管理できる特別権限 | 組織設定、アカウント・権限管理、マスタ設定、緊急保守。管理能力自体は新システムでも必要 | 共用アカウントとしては引き継がない。個人を識別できる管理者アカウント+通常権限と緊急時の一時昇格の分離 |
| 経理専用の特別権限 | 売上・精算・会計系の情報を店舗の日常業務・公開コンテンツ管理から分離する。閲覧できる会計範囲を通常の店舗範囲と別に扱える構造も確認済み | 「特別アカウント」ではなく、会計機能と対象範囲を組み合わせた権限として実装 |
| 監査向けの一時的な表示制限 | 特定のログイン時に受注・金額・個人情報の一部を通常より少なく表示・伏せる仕組み。現在も実際の監査・確認業務で使われているかはコードから判断できない | 業務担当者が利用場面を確認できた場合に限り「対象範囲と期間を限定した、伏せ字付きの閲覧専用権限」として残す（採否は業務確認待ち） |
| 特定利用者だけが使える機能や省略できる確認 | 個別利用者を条件にした例外は確認できるが、業務理由・責任者・有効期限はコードに記録されていない — 必要性を確認できない | そのまま移さない。業務上の理由・承認者・対象期間が確認できた例外のみ、正式な権限として再設計 |
| ログイン後に決められた画面へ移動する設定（既定画面） | 利用者の種類により予約・受注一覧や会計画面へ直接移動する。権限ではなく操作を早くするための初期画面設定 | 利用者またはグループごとの既定画面設定として残せる。データへのアクセス可否とは分離する |
| 外部サービス連携専用のアカウント | 外部媒体への自動ログイン・投稿・情報取得に専用の認証情報が必要。外部連携を継続する間は用途自体が必要 | 人がログインする共用アカウントにせず、用途・接続先・利用可能な処理を限定したサービス ID として管理し、認証情報を定期的に更新できる設計にする |

結論（移行元提案）: 業務能力として残す必要が高いのは「全体管理」「経理」「外部連携」。「監査向け表示制限」は実利用の確認後に判断。「個人別の例外」は理由が確認できない限り引き継がない。「既定画面」は権限ではなく設定として扱う。

引き継ぎ方の共通原則（移行元提案 — #368 裁定済み）:

- 旧アカウント名・利用者 ID を権限として使わず、必要な能力へ分解する（履歴閲覧・承認・会員サポート・経理・担当店舗・承認済みデータのみの閲覧・既定画面 等）。
- 全体管理は「個人を追跡できる管理者権限」と「緊急管理権限」に分ける。緊急管理は利用者と利用日時を追跡する。
- 経理は独立した能力として扱う。
- 定期処理・外部連携は人の共有アカウントではなく、用途と対象範囲が明確なサービス ID として扱う。
- **業務理由を確認できない個人別例外は引き継がない。**

## 本人ログイン

- キャストは通常、本人用 ID とパスワードでログインする。限定された導線では電話番号とパスワードも使われる。通常認証とは異なる古い経路が残っている（ここまで確認済み）。**古い経路は Kizuna へ引き継がず、セキュリティ確認が必要（移行元提案 — #368 で裁定）。**
- 会員は電話番号とパスワードでログインする。LINE は連絡先案内であり、LINE 認証ではない（確認済み）。
- 利用者が認識している ID・電話番号との連続性は必要。パスワード方式と古いログイン経路はそのまま引き継がない（移行元提案 — #368 で裁定）。
- Kizuna 既定: 会員登録はメール + 会員コード発行（#331）、LINE ログインはプラットフォーム認証手段として実装予定（#332）— 現行に LINE 認証はないため新規要件として扱う（回答でも同じ整理）。

## 権限の付与・変更・停止

- 付与・変更・停止は「組織・システム管理」能力の業務（上表参照）。
- 退職・異動・契約終了・店舗閉鎖時にアクセスを停止しても、過去の実行主体の記録は保持する（#364 本文の要件。現行は監査履歴が全データにあるわけではない — data-model.md 条件 8）。
- 実運用上の責任者（誰が実際に付与を承認するか）は業務確認待ち。

## 業務担当者への確認が必要な事項（まとめ）

2026-07-17 の spec 化前 Q&A で、特別アカウント 6 種の業務理由・緊急管理追跡の現状（満たしていない）・操作単位・広告費/応募者情報の帰属は本文へ反映済み。回答者による現行コードとの再照合は実施済みだが、現場担当者による確認は未実施のため、次を業務確認待ちとして残す:

- 実際に利用中の社内利用者数と本社・店舗・経理別の内訳（アカウント移行前の匿名集計）
- 実際の兼務パターン（誰がどの業務を兼ねているか、その頻度）
- 経理担当者と広告担当者が実際に同一人物か（独立能力としての扱いは移行元提案 — #369 候補 4 / #382）
- 応募者情報の状態別の保存・削除方針と、選考の最終責任者（workflows.md フロー 15 と共通）
- 監査向けの一時的な表示制限の実際の利用場面（確認できた場合に限り閲覧専用権限として採用）
- 権限の付与・変更を実際に承認している責任者
- 店舗業務・キャスト対応・会員対応を知る担当者による本書の内容確認（#364 の受け入れ条件、未実施 — #364 コメントの後続 Q&A で依頼する）
