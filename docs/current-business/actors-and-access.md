# 現行業務の利用者とアクセス範囲

現行業務システムの業務を Kizuna が引き受けるための材料として、利用者種別・機能権限・データ範囲を業務語彙のみで記録する。出典は #364（本文と回答コメント）。移行ロードマップ地図 #367 の材料であり、裁定票 #368 の入力になる。

方針（#367 の Notes に準拠）:

- 旧システムのアカウント名・数値権限・利用者 ID・実装方式は記録しない。「誰が何をできるか」と「どの範囲のデータを扱えるか」だけを扱う。
- 本質を先に、細部は後で。詳細は裁定後の追加 Q&A で補う。
- 業務の流れは workflows.md（#362）、データは data-model.md（#363）が持つ。本書は利用者と権限だけを持つ。
- #320 / #321 / #328〜#336 / #339 / #352 の実装方針は上書きしない。本書は業務事実の提供のみ。

最終更新: 2026-07-16（#364 の回答時点）

## 規模感（匿名化テストデータの参考値）

- 有効設定のある社内利用者 37 件、うち 25 件は複数店舗を担当（確認済み）。
- 実際に利用中の人数、本社・店舗・経理別の内訳は業務確認待ち。権限設計は人数確定を待つ必要はないが、アカウント移行前に本番の利用者一覧の匿名集計が必要。

## 権限モデルの本質（確認済み）

通常の社内利用者の利用範囲は、次の組合せで決まる。

- 利用できる機能（機能権限）
- 担当店舗
- 必要な場合だけ精算対象店舗
- アカウントの有効・無効状態

利用者・能力の一覧は #364 本文の表を正とする（10 の利用者・能力 × 主な業務 × 扱うデータ × データ範囲）。固定された 10 個のロールを作るのではなく、**「一人の社内 ID + 機能権限 + 担当店舗 + 必要な場合だけ精算範囲（+ 外部サービスの対象範囲）」**で表現するのが現行業務に最も近い形（確認済み）。

### 能力の統合可否（確認済み）

- 店舗管理・受付と当日店舗運営は同じ社内 ID で利用できる。ただし機能権限は分ける。
- キャスト登録・公開情報管理・写真付き投稿・外部サービス運用は同じ担当者へ複数能力を付与できる。ただし内部情報と公開操作の権限は分ける。
- 顧客・予約・受注の店舗業務は、担当店舗の範囲と組み合わせて一つの社内 ID にまとめられる。
- 経理・精算は通常の店舗操作とは別能力にし、対象となる精算単位を限定する。
- システム管理・緊急管理は通常業務から分ける。
- キャスト本人・会員本人・一般来訪者は、社内利用者とは別の本人種別として扱う。
- 実際の兼務パターン（誰がどの業務を兼ねているか）は業務確認待ち。

## 特別アカウントの棚卸し

現行の例外（確認済み）:

- 全体を管理できる特別権限
- 経理専用の特別権限
- 監査向けの一時的な表示制限
- 特定利用者だけが使える機能や省略できる確認
- ログイン後に決められた画面へ移動する設定（既定画面）
- 外部サービス連携専用のアカウント

引き継ぎ方（移行元提案 — #368 で裁定）:

- 旧アカウント名・利用者 ID を権限として使わず、必要な能力へ分解する（履歴閲覧・承認・会員サポート・経理・担当店舗・承認済みデータのみの閲覧・既定画面 等）。
- 全体管理は「個人を追跡できる管理者権限」と「緊急管理権限」に分ける。緊急管理は利用者と利用日時を追跡する。
- 経理は独立した能力として扱う。
- 定期処理・外部連携は人の共有アカウントではなく、用途と対象範囲が明確なサービス ID として扱う。
- **業務理由を確認できない個人別例外は引き継がない。**

## 本人ログイン（確認済み）

- キャストは通常、本人用 ID とパスワードでログインする。限定された導線では電話番号とパスワードも使われる。**通常認証とは異なる古い経路が残っており、Kizuna へは引き継がずセキュリティ確認が必要。**
- 会員は電話番号とパスワードでログインする。LINE は連絡先案内であり、LINE 認証ではない。
- 利用者が認識している ID・電話番号との連続性は必要。パスワード方式と古いログイン経路はそのまま引き継がない。
- Kizuna 既定: 会員登録はメール + 会員コード発行（#331）、LINE ログインはプラットフォーム認証手段として実装予定（#332）— 現行に LINE 認証はないため新規要件として扱う（回答でも同じ整理）。

## 必ず確認するアクセス場面（8 場面）と受け皿

8 場面の正本は #364 本文とする。各場面の「期待される許可・拒否」は本文の記述そのものであり、ここでは現行実態と Kizuna 側の受け皿だけを対応付ける。

| # | 場面（要旨） | 現行実態（確認済み） | Kizuna 側の受け皿 |
|---|---|---|---|
| 1 | 一店舗担当者は他店舗のデータを閲覧・更新できない | 機能×担当店舗の組合せで実現 | 既定: テナント/店舗スコープ（#339 の集合作用域ガード含む） |
| 2 | 複数店舗担当者は店舗別と集約の表示を使い分け | 25/37 件が複数店舗担当 | 既定: プラットフォーム身分 + 店舗集合授権（#320） |
| 3 | 経理権限は店舗操作と分離し精算単位を限定 | 経理専用の特別権限が存在 | 金流域ごと #369 で裁定 |
| 4 | キャストは自分の情報・割当受注のみ | 本人種別として分離 | 既定: キャストポータル（#320/#321） |
| 5 | 会員は本人の予約・履歴・ポイントのみ | 本人種別として分離 | 既定: 会員ポータル（#320/#321、#336） |
| 6 | 公開情報に内部情報が混入しない | 分離されているが同義値が複数箇所（data-model.md 条件 5） | #368 で正本と境界を裁定 |
| 7 | 定期処理・外部連携は対象範囲と結果を追跡 | 連携用アカウントに表示上の例外あり（data-model.md 条件 6） | サービス ID 化の採否を #368、処理自体は #370 で裁定 |
| 8 | 緊急管理権限の利用者・日時・操作を追跡 | 全権特別権限は存在するが個人追跡の分離はない | 管理者/緊急管理の分離提案を #368 で裁定 |

## 権限の付与・変更・停止

- 付与・変更・停止は「組織・システム管理」能力の業務（#364 本文の表）。
- 退職・異動・契約終了・店舗閉鎖時にアクセスを停止しても、過去の実行主体の記録は保持する（#364 本文の要件。現行は監査履歴が全データにあるわけではない — data-model.md 条件 8）。
- 実運用上の責任者（誰が実際に付与を承認するか）は業務確認待ち。

## 業務担当者への確認が必要な事項（まとめ）

- 実際に利用中の社内利用者数と本社・店舗・経理別の内訳（アカウント移行前の匿名集計）
- 実際の兼務パターン
- 特別アカウント個々の業務理由と要否（「業務理由を確認できない個人別例外は引き継がない」方針の適用対象を確定するため）
- 監査向けの一時的な表示制限の要否
- 権限の付与・変更を実際に承認している責任者
- 店舗業務・キャスト対応・会員対応を知る担当者による本書の内容確認（#364 の受け入れ条件、未実施 — #364 コメントの後続 Q&A で依頼する）
